# 镜像多维度空间（Mirror Multi-Dimensional Space）
开发者：自由的风

本规格定义“镜像多维度空间”作为所有 Agent 的编程、修复、部署必经之路的强制门禁体系：不可跳步；默认一条龙真实执行（单节点 Canary + 自动回滚）；生产面板只允许生产级代码（除 `*.md` 外严格阻断）；并集成计费与设计大脑（Design Brain）。

---

## 0. 范围与术语

### 0.1 范围
- 适用：任何 Agent 触发的“修复/部署/回滚/发布”链路；任何 CI/CD 的生产发布链路；任何开发者本地发布链路
- 交付形态：Public 可下载二进制（不公开源码），Private 保持源码；云端提供策略与契约判定、签名票据、审计与计费联动

### 0.2 核心术语
- 模块（Module）：项目中可独立镜像、测试、制品化、替换的单元（Node/Python/混合）
- 镜像空间（Mirror Space）：对目标模块的隔离副本，仅在此空间允许写修复代码与验证
- 生产面板（Production Panel）：真实线上代码目录/部署目录
- 纯度（Purity）：生产面板只允许生产级代码，禁止占位/空壳/假数据/调试代码/硬编码凭据
- Gate（门禁）：任何不满足条件即阻断生产发布的强制检查点

---

## 1. 铁律（Hard Gates）

### 1.1 不可跳步执行
- 全链路必须按“假设 → 理论 → 设计 → 架构 → 编码 → 测试 → 验证 → 发布”执行
- 未生成设计蓝图、未确认设计蓝图、未获得票据，禁止进入下一阶段

### 1.2 生产纯度 Hard Gate（仅豁免 `*.md`）
- 生产面板、将要进入生产的制品（artifact）中，除 `*.md` 外一律严格扫描并阻断：
  - 占位：`placeholder`、`FIXME`、`TODO-fix`、`xxx` 等
  - 空壳：`pass`、`return null`、`not implemented` 等
  - 假数据：`fakeData`、`dummyData`、`mockData`、`sample data`、`假数据` 等
  - 调试代码：`debugger;`、`console.log`、`print(` 等
  - 硬编码凭据：形如 `sk-...`、`api_key="..."`、`password="..."`、token 等
- 任何命中：直接阻断生产发布；不得降级为 warning

### 1.3 云端不可用禁止生产发布
- 策略包与契约兼容判定、票据签名均由云端执行
- 云端不可达或验签失败：允许本地预检/演练，但禁止生产 deploy

### 1.4 只允许不可变制品 + 一次性原子切换
- 禁止边跑边覆盖生产目录
- 发布必须基于不可变制品（artifact），并以一次性切换（symlink/junction/等价原子策略）完成
- 任何切换必须具备回滚点（旧制品指针 + 数据快照）

### 1.5 灰度与快速回滚（默认单节点）
- 默认 Canary 策略：单实例/单节点先行验证
- 任何阶段失败：自动回滚到旧制品，并按数据策略回滚/恢复快照

---

## 2. 三步原子替换修复链路（强制）

### 2.1 环境一致（标准化 + 自动化 + 版本锁定）
- 同一套基线：运行时、包管理器、系统依赖的可比对清单
- 同一份配置模板：敏感信息仅允许变量/密钥管理注入
- 自动化拉起环境：bootstrap 脚本/流水线替代人工修改配置
- 锁定：运行时版本、库版本、系统包版本；生成环境指纹与差异报告

### 2.2 数据一致（迁移成对 + 幂等 + 快照）
- 所有结构/数据变更必须走迁移脚本，禁止生产手工改表/改数据
- 迁移必须成对：`up` 与 `down`
- 迁移必须幂等：重复执行不可产生破坏性副作用
- 执行前必须快照/备份；失败或校验异常可回滚

### 2.3 调整与校验（可观测 + 对比 + 演练）
- 发布前：与生产同构的预发做完整演练，包含回滚演练
- 发布后：数据校验（行数/校验和/核心指标对比）+ 日志/监控；异常尽早回滚

---

## 3. “必经之路”门禁体系（不可绕过）

### 3.1 三重门禁（Agent + CI + 本地）
- Agent 门禁：任何 Agent 的 deploy/repair/release/rollback 意图必须先调用 `jx pipeline` 与 `jx design`
- CI 门禁：生产 deploy job 首行强制执行 `jx pipeline`；并要求有效票据
- 本地门禁：pre-push/pre-tag/release 钩子阻断直改生产面板与无票据发布

### 3.2 通行证机制：没有票据就不允许发布
- `design.ticket`：证明“设计蓝图已付费生成并被用户确认”
- `deploy.ticket`：证明“某模块已通过所有 Gate 且计费预授权有效，允许发布”
- CI 与本地门禁必须验签；验签失败或过期即阻断

---

## 4. 计费与报价（不可跳过）

### 4.1 免费 Quote（报价阶段）
- Quote 阶段免费：扫描项目并输出本次执行将产生的调用次数与总费用
- 计费单位（执行）：每模块一次性执行计费 1 美元（修复或部署都算一次）
- 计费单位（设计）：每次生成一份设计蓝图计费 1 美元

### 4.2 报价输出（必须可解释）
- 输出内容：
  - 模块清单（将执行哪些模块）
  - 次数：`N = 模块数量`
  - 总价：`N * $1`
  - 设计蓝图次数（默认 1 份；若拆分为多份蓝图则单独列出）

### 4.3 发现新增模块：立即暂停二次报价二次确认
- 执行过程中如果发现需要新增模块（导致 N 增加）：
  - 立即暂停
  - 生成新的 Quote
  - 用户二次确认后才能继续执行

### 4.4 扣费策略（预授权）
- 设计蓝图：用户确认生成蓝图前预授权 1 美元；生成成功后结算；失败释放
- 模块执行：模块开始前预授权 1 美元；模块成功（通过验证并完成切换）结算；失败/回滚释放

---

## 5. 设计大脑（Design Brain）

### 5.1 定位
- Design Brain 是“必经之路”的第一环：先输出专业设计蓝图与说明，用户确认后才能执行

### 5.2 输出模板（C：混合模板，必须包含）
- 假设与边界（目标/约束/风险）
- 理论与铁律（Hard Gates）
- 系统架构蓝图（组件图/数据流/状态机）
- 执行设计（门禁/票据/审计/回滚矩阵）
- 报价与计费说明（用户可理解）
- 可观测与校验（数据校验项与阈值）
- 产物清单（机器可读）

### 5.3 票据绑定
- 未获得且未确认 `design.ticket`：禁止签发 `deploy.ticket`
- `deploy.ticket` 必须引用 `design_ticket_id` 与 `design_hash`

---

## 6. Pipeline（默认真实执行，一条龙）

### 6.1 默认模式
- 默认真实执行
- 默认 Canary：单实例/单节点
- 失败自动回滚（制品切回 + 数据回滚/快照恢复）

### 6.2 强制阶段（不可跳过）
- Purity Gate（严格，除 `*.md` 外无豁免）
- Env Consistency（锁版本 + 自动化 bootstrap + 环境指纹）
- Data Consistency（迁移 up/down + 幂等 + 快照 + 回滚演练策略）
- Contract Gate（本地提取 → 云端判定；云端不可用阻断）
- Canary（单节点）→ Smoke Test → Verify（数据校验/指标对比）
- Promote（不可变制品 + 一次性原子切换）→ Post-Verify → Monitor

---

## 7. 契约与数据兼容（扎实设计）

### 7.1 契约稳定（Contract Gate）
- 本地提取契约候选：exports、事件、依赖图、关键配置面（可扩展）
- 云端做兼容判定：移除 export/事件、破坏性变更等一律阻断
- 输出 `contract_hash`，写入票据与审计

### 7.2 数据兼容（Data Gate）
- 迁移规则与破坏性变更检测（可配置）
- 旧写新读/新写旧读风险隔离策略（以规则阻断为 V1 主策略）
- 关键数据必须快照/备份并可恢复

---

## 8. 票据与验签（云端私钥，本地/CI 公钥）

### 8.1 design.ticket（云端签名）
- 目的：证明“蓝图已生成且用户确认”
- 必须字段（示例）：
  - `design_ticket_id`

# 镜像多维度空间（Mirror Multi-Dimensional Space）
开发者：自由的风

本规格定义“镜像多维度空间”作为所有 Agent 的编程、修复、部署必经之路的强制门禁体系：不可跳步；默认一条龙真实执行（单节点 Canary + 自动回滚）；生产面板只允许生产级代码（除 `*.md` 外严格阻断）；并集成计费与设计大脑（Design Brain）。

---

## 0. 范围与术语

### 0.1 范围
- 适用：任何 Agent 触发的“修复/部署/回滚/发布”链路；任何 CI/CD 的生产发布链路；任何开发者本地发布链路
- 交付形态：Public 可下载二进制（不公开源码），Private 保持源码；云端提供策略与契约判定、签名票据、审计与计费联动

### 0.2 核心术语
- 模块（Module）：项目中可独立镜像、测试、制品化、替换的单元（Node/Python/混合）
- 镜像空间（Mirror Space）：对目标模块的隔离副本，仅在此空间允许写修复代码与验证
- 生产面板（Production Panel）：真实线上代码目录/部署目录
- 纯度（Purity）：生产面板只允许生产级代码，禁止占位/空壳/假数据/调试代码/硬编码凭据
- Gate（门禁）：任何不满足条件即阻断生产发布的强制检查点

---

## 1. 铁律（Hard Gates）

### 1.1 不可跳步执行
- 全链路必须按“假设 → 理论 → 设计 → 架构 → 编码 → 测试 → 验证 → 发布”执行
- 未生成设计蓝图、未确认设计蓝图、未获得票据，禁止进入下一阶段

### 1.2 生产纯度 Hard Gate（仅豁免 `*.md`）
- 生产面板、将要进入生产的制品（artifact）中，除 `*.md` 外一律严格扫描并阻断：
  - 占位：`placeholder`、`FIXME`、`TODO-fix`、`xxx` 等
  - 空壳：`pass`、`return null`、`not implemented` 等
  - 假数据：`fakeData`、`dummyData`、`mockData`、`sample data`、`假数据` 等
  - 调试代码：`debugger;`、`console.log`、`print(` 等
  - 硬编码凭据：形如 `sk-...`、`api_key="..."`、`password="..."`、token 等
- 任何命中：直接阻断生产发布；不得降级为 warning

### 1.3 云端不可用禁止生产发布
- 策略包与契约兼容判定、票据签名均由云端执行
- 云端不可达或验签失败：允许本地预检/演练，但禁止生产 deploy

### 1.4 只允许不可变制品 + 一次性原子切换
- 禁止边跑边覆盖生产目录
- 发布必须基于不可变制品（artifact），并以一次性切换（symlink/junction/等价原子策略）完成
- 任何切换必须具备回滚点（旧制品指针 + 数据快照）

### 1.5 灰度与快速回滚（默认单节点）
- 默认 Canary 策略：单实例/单节点先行验证
- 任何阶段失败：自动回滚到旧制品，并按数据策略回滚/恢复快照

---

## 2. 三步原子替换修复链路（强制）

### 2.1 环境一致（标准化 + 自动化 + 版本锁定）
- 同一套基线：运行时、包管理器、系统依赖的可比对清单
- 同一份配置模板：敏感信息仅允许变量/密钥管理注入
- 自动化拉起环境：bootstrap 脚本/流水线替代人工修改配置
- 锁定：运行时版本、库版本、系统包版本；生成环境指纹与差异报告

### 2.2 数据一致（迁移成对 + 幂等 + 快照）
- 所有结构/数据变更必须走迁移脚本，禁止生产手工改表/改数据
- 迁移必须成对：`up` 与 `down`
- 迁移必须幂等：重复执行不可产生破坏性副作用
- 执行前必须快照/备份；失败或校验异常可回滚

### 2.3 调整与校验（可观测 + 对比 + 演练）
- 发布前：与生产同构的预发做完整演练，包含回滚演练
- 发布后：数据校验（行数/校验和/核心指标对比）+ 日志/监控；异常尽早回滚

## 3. “必经之路”门禁体系（不可绕过）

### 3.1 三重门禁（Agent + CI + 本地）
- Agent 门禁：任何 Agent 的 deploy/repair/release/rollback 意图必须先调用 `jx pipeline` 与 `jx design`
- CI 门禁：生产 deploy job 首行强制执行 `jx pipeline`；并要求有效票据
- 本地门禁：pre-push/pre-tag/release 钩子阻断直改生产面板与无票据发布

### 3.2 通行证机制：没有票据就不允许发布
- `design.ticket`：证明“设计蓝图已付费生成并被用户确认”
- `deploy.ticket`：证明“某模块已通过所有 Gate 且计费预授权有效，允许发布”
- CI 与本地门禁必须验签；验签失败或过期即阻断

## 4. 计费与报价（不可跳过）

### 4.1 免费 Quote（报价阶段）
- Quote 阶段免费：扫描项目并输出本次执行将产生的调用次数与总费用
- 计费单位（执行）：每模块一次性执行计费 1 美元（修复或部署都算一次）
- 计费单位（设计）：每次生成一份设计蓝图计费 1 美元

### 4.2 报价输出（必须可解释）
- 输出内容：
  - 模块清单（将执行哪些模块）
  - 次数：`N = 模块数量`
  - 总价：`N * $1`
  - 设计蓝图次数（默认 1 份；若拆分为多份蓝图则单独列出）

### 4.3 发现新增模块：立即暂停二次报价二次确认
- 执行过程中如果发现需要新增模块（导致 N 增加）：
  - 立即暂停
  - 生成新的 Quote
  - 用户二次确认后才能继续执行

### 4.4 扣费策略（预授权）
- 设计蓝图：用户确认生成蓝图前预授权 1 美元；生成成功后结算；失败释放
- 模块执行：模块开始前预授权 1 美元；模块成功（通过验证并完成切换）结算；失败/回滚释放
