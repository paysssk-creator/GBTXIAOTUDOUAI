# 工程师工作铁律 — Engineering Iron Law

> 适用于 GBT 项目组所有人（人类 + AI 伙伴）。写给小土豆和每一个参与项目的人。

---

## 五条铁律

```
1. 看清楚
   了解项目结构、文件关系、改动范围。
   不扫全局不撒手，不确认边界不动刀。

2. 规划好
   明确怎么做、影响哪些文件、不碰哪些。
   先看有没有现成工具能直接用，别重复造轮子。

3. 动手前
   确认一遍方案合理，确认两遍不会搞混项目。
   镜像空间审计跑一遍，知道基线在哪再改。

4. 做完后
   检查三遍：
     a) 能不能用？（用户视角走一遍）
     b) 会不会影响别的？（跑全量测试）
     c) 提交了没？（git add + commit + push 缺一不可）

5. 交付前
   下载 → 安装 → 使用 → 测试 → 直到发现不了问题为止。
   用户用得不安心，就是我们的错。
```

---

## 测试标准

- **不通过不交付**：任何命令报错、任何模块 import 失败，必须修完再交付
- **覆盖三视角**：用户视角（能用）、运维视角（可维护）、业务视角（功能对）
- **发现问题继续测**：修完一个继续跑，直到全部通过，不是修完就停

---

## 禁止事项

- ❌ 没看清就动手
- ❌ 做完了不 commit
- ❌ 跳过测试直接交付
- ❌ 搞混项目边界
- ❌ 重复造轮子（先看看有没有现成的）

---

> 记住：影响用户使用心情，就是我们的错。
